Shell Basics

First thing is first. Lets figure out where you are at!

pwd C/Users/Alex/

This just printed your current directory. If you just opened your terminal window this should be your home directory

Lets figure out what is in this directory

ls

ls will print the contents of your directory. There are some files that aren't shown. To see these files we can add arguments to our commands

For example we can list all the contents of this directory with

ls -a

The first two directories here are special paths that are in every directory

. is the current directory, so to access a hidden file in the directory you would type in .file

.. is the previous directory

Moving Around

cd is how we Change Directories

cd .. moves up one directory so if you are in /c/git/ you will go up a directory to /c/

When filling out paths in the shell you can hit the tab key at any point to attempt to auto complete the path

For example cd /c/git/HO/HO.Wetab will auto complete to cd /c/git/HO/HO.Weforms/

Create directories and files

touch fileName will create a file called fileName

mkdir directory creates a directory named directory

Remember directory and file names are case sensitive

File Manipulation

we can move files with mv {fileToMove} {new/path/to/fileToMove}

copy files with cp {file} {copiedFile}

to remove a file use rm {fileName}

Getting help

If you need help with any command you can access the help menu for most commands with {Command Name} --help

For more information you can open the manual for the command with man {Command Name}

Don't forget to try google. There are many useful references online. We are using Bash (Born Again SHell)

rm -rf {directory}removes a full directory. Be careful with this one because there isn't an easy way to get your stuff back

Git Basics

Git is the source control program we use. It is a very powerful tool set that can be used for any project. It allows for easy file revisions and collaboration between many people

To start navigate to the project you have done work in and identify your issue number

We need to make a branch before we start coding so our work can be submitted and reviewed separately from our working branch which is usually develop

First Time Configuration

We have to setup your name, email, and push behavior

git config --global user.name "John Doe"

git config --global user.email johndoe@example.com

git config --global push.default simple

Branch Naming

For issue number that are 6 places like issue #123456 we use the scheme SP/123000/400/123456

For issue number with 5 places like #12345 we use SP/12000/12345

For features that don't have an issue number feature/{FirstInital}{LastName}/{distinctFeatureName}

Creating and moving to the branch

git branch {branch name} Will create your branch

To move to that branch so you can start coding git checkout {branch name}

Now that we have our branch go write some code and come back after you have made some progress

Committing your code

A commit is how we creates a checkpoint for our code. It saves all the work you have done.

Commits should be done early and often. Think every time you come to a good stopping point.

Commits can also be done to just save your work at the end of the day or whenever you need to work on something else

The first part of the commit is to see the status of your work. For this example we are going to be using the directory of the project this file is in.

git status 
On branch master
	Your branch is up-to-date with 'origin/master'.

	Untracked files:
	  (use "git add <file>..." to include in what will be committed)

		MovingAroundTheShell.html
		main.css
		scripts/

	no changes added to commit (use "git add" and/or "git commit -a")

Here we can see many different things that were output

To start we see

On branch master
	Your branch is up-to-date with 'origin/master'.

This is the the current branch you are on. Yours should correspond to the issue you are working on.

Untracked files:
	  (use "git add <file>..." to include in what will be committed)

		MovingAroundTheShell.html
		main.css
		scripts/

These are new files that were added and are untracked by git

We need to add them with git add {filename} or if you want to add them all at once you can use

git add --all

Now to see what happened we can use

git status 
On branch master
	Your branch is up-to-date with 'origin/master'.

	Changes to be committed:
	  (use "git reset HEAD <file>..." to unstage)

		new file:   MovingAroundTheShell.html
		new file:   main.css
		renamed:    git-safeRebase -> scripts/git-safeRebase

We see that the files are staged and ready to be committed. If you want to remove a single file you can use git reset HEAD <file>

To reset all of the file you can use git reset HEAD --mixed

HEAD is the current state of your branch

Be very careful with the reset command. You can lose work if you use it wrong

git commit

This launched the text editor VIM by default

To write your commit message you must first press the a (amend) or the i (insert) key

Now you can type your commit message out as normal. When you are done hit esc and then type :wq and hit enter to save your commit message

Getting your code ready to push

Now that you have completed your code you must get it ready to push to the remote

Since we want to keep our git history nice and clean you should flatten your branch by resetting all the commits on it

To find out how many commits are on your branch you can use git cherry develop | wc -l remember this number for later.

This will spit out the number of commits you have. It is a combination of two commands piped with the output of git cherry develop piped into wc -l

git cherry develop compares your branch to develop and will show you the ids of all your commits.

wc -l counts the lines in taken from an input. This is a generic tool you can use anywhere.

With the number from above you can now reset the branch with git reset HEAD~{number from earlier}

Now restage and commit your code

Rebasing

Now we need to get our code up to date with the current state of develop

We do this with a rebase

First thing is first. Chekout to develop with git checkout develop and then pull to get latest git pull

Now check back out to the branch you were working on git checkout {branchName}

git rebase develop will move your branch forward in time so it includes the changes in the most recent code

You can do this at anytime especially when you need new features that have been merged into develop.

There might be merge conflicts. Don't be worried just go find each one and fix it. Once done restage your files with git add --all and then continue the rebase with git rebase --continue

DO NOT rebase if your code has already been pushed to develop

Pushing your code to origin

It is time to get our code onto the remote server

We do this with git push -u the -u is only needed the first time you push since its sets up tracking from the upstream branch
